深入探索SparkSql的优化器Catalyst

SparkSql 是最新以及最有技术含量的Spark组件之一。由SQL查询和新的DataFrame API组成。SparkSQL的核心是优化器Catalyst，利用高级编程语言的特征（Scala的pattern matching 和 quasiquotes）以一个不同的方式来建立一个可扩展的查询优化器。

我们最近在SIGMOD 2015上发布一篇关于Spark SQL的论文。这篇博客我们再发一次论文中解释Catalyst优化器内部原理的部分。

我们设计了一个可扩展的新的优化器Catalyst来实现SparkSQL，它基于scala函数式编程理念。Catalyst的可扩展设计有两个目的。首先，我们想容易地为SparkSQL增加新的优化技术和特征，尤其是想要处理我们在大数据量时看到的各种各样的问题（半结构化数据和高级分析）。其次，我们想让外面的开发者也参与优化器的扩展——比如，通过增加数据源指定规则来推动i过滤（filtering）或聚合（aggregatioln）进入外部存储系统，或者支持新的数据类型。Catalyst支持基于规则和基于代价的优化。

在core模块中，Catalyst包含一个外部库来表示树结构以及操作它们的应用规则。在这个框架的顶部，我们针对相关查询进程（表达式，逻辑查询计划等等）建立一些特殊的库，以及执行查询中的处理不同阶段的几个规则，这些阶段包括分析（analysis），逻辑优化（logical optimization），物理计划（physical planning）和代码生成即编译查询的各个部分为java字节码。之后，我们使用另一个scala特征，quasiquotes，使得运行时从组合的表达式中产生代码变得简单。最后，Catalyst提供几个公有的扩展点，包括外部数据源和用户自定义类型。

Trees

Catalyst中主要的数据类型是一棵由节点对象组成的树。每一个节点有一个节点类型和零个或多个的孩子。使用scala来声明新的节点类型为TreeNode类的子类。这些对象是不可变的，并且能够使用函数式转换来操作，下个小节会讲到。

一个简单的例子，假设我们有以下三个节点类作为简单的表达式语句：

· Literal（value : Int）：一个常量

· Attribute（name : String） : 一个输入行的属性，例如“X”

· Add（left : TreeNode, right : TreeNode）: 两个表达式的和

使用Rules来操作Trees，Rules是一些函数表示从一棵树对别的树的操作的集合。当一个规则能直接运行代码于它的输入树（已知这树是一个Scala对象），最普遍的方法是使用一系列的模式匹配函数来找到子树并以一个特定的结构替换之。

模式匹配表达式被转换成等价的函数形式，就是说它仅仅需要匹配的是所有可能输入树的一个子集。Catalyst将测试一棵树中 适用给定规则的那些部分，如果没有匹配上，会自动跳过并递归下探到子树中。这个能力意味着，规则只需要去推断给定优化适用的树，没匹配上的就不需要了。因此，规则不需要被修改成一种新的运算类型加入系统中。

最后，规则条件和它们的主体能包含任意Scala代码。这就在优化器上使Catalyst比其他领域特定语言更强大，然而对简单的规则又保持简单。

我们的经验中，对不可变的树进行函数式转换，令整个优化器非常容易地去推导和调试。尽管我们没有开发这个，在优化器中并行进行转换也因此成为可能。

Using Catalyst in Spark SQL

我们在以下四个阶段中使用了Catalyst中普通的树转换框架，（1）分析逻辑计划来决定引用，（2）逻辑计划的优化，（3）物理计划，以及（4）代码生成以编译查询的各部分为java字节码。在物理计划阶段，Catalyst可能产生多个计划，然后基于代价进行比较。其他所有阶段是单纯基于规则的。每个阶段使用不同的树节点类型；Catalyst包括各种表达式、数据类型、逻辑操作和物理操作的节点库。我们描述的各个阶段如下：

Analysis

Spark SQL以关系计算为开端，由SQL分析器（parser）返回的抽象语法树（AST），或使用API构造的DataFrame对象，都是如此。这两种情况下，关系可能包含未解决的属性引用或关系：例如，SQL查询中SELECT col FROM sales，列的类型，或者甚至是否合法的列名，直到我们找到这个sales表之前都是未知的。一个属性，如果我们不知道它的类型或没有匹配上任意一个输入表（或别名），那么这个属性将被叫做“unresolved”。SparkSQL使用了Catalyst的规则和用于跟踪所有数据源的表来决定这些属性的一个Catalyst对象。由无约束的属性和数据类型开始建立一个“unresolved logical plan”树，然后使用规则完成下面的事：

· 通过目录中的名字查找对应关系。

· 将有名字的属性，如列，映射到提供了特定操作器或其子类的输入。

· 决定哪些属性指向同一个值，并给它们一个唯一ID（为了之后优化表达式，如col = col）。

· 通过表达式来传播和支配数据类型：如，直到我们清晰了列然后强制将它的子表达式转成一个兼容的类型，我们才能知道1+col的返回值类型。

Physical Planning

在物理计划阶段，SparkSQL使用匹配Spark执行引擎的物理操作器，对一个逻辑计划产生一个或多个物理计划。然后它根据代价模型选择一个计划。此时，基于代价优化仅被用来选择join算法：对于那些已知较小的关系，SparkSQL用broadcast join，其用到Spark中的P2P对等广播技术。这个框架支持基于代价优化的广泛用途，然而，对于使用了规则的一棵树来说，代价总能被递归地估算出来。因此我们将来打算去实现更加丰富的基于代价的优化。

物理计划也执行基于规则的物理优化，如并行地投影或过滤包含进一个Spark的map操作中。例外，它能把逻辑计划中的操作推到 支持谓词或投影下推的数据源。我们将在稍后的篇章讲述这些数据源。

总计，物理计划大概500行代码。

Code Generation

查询优化的最后一个阶段是关于产生java字节码在每个机器上运行。因为SparkSQL经常在内存中操作数据集，处理的是计算密集型（CPU-bound）的，我们希望它能支持代码生成以加速执行。尽管如此，代码生成引擎的构建通常是很复杂的，本质上相当于一个编译器。Catalyst依靠一个scala语言的一个特征，quasiquotes，来让代码生成更简单。Quasiquotes允许抽象语法树（AST）的大纲式结构，这种结构可以满足scala编译器在运行时产生字节码。我们使用Catalyst来转换一颗树，表示一个SQL表达式转换成一个scala代码的AST并估算表达式的值，然后编译和运行产生的代码。

一个简单的例子，试考虑在Section 4.2中提到的Add，Attribute和Literal树节点，能写成形如（x+y）=1 的表达式。没有代码生成，这样的表达式必须为数据中的每一行进行翻译，而遍历Add，Attribute和Literal节点这棵树。这就引进了大量的分支和虚函数调用使执行变得缓慢。有了代码生成，我们可以写一个函数来翻译某个特定的表达式树 成scala AST：

q打头的字符串就是quasiquotes，意味着尽管它们看起来想字符串，它们在编译时被scala编译器编译，代表着包含代码的AST。Quasiquotes可以包含变量或者叠加其他AST，都用”$”来表示。例如Literal(1)会变成 1 的scala AST，而Attribute(“x”)变成row.get(“x”)。最后，一棵形如Add(Literal(1), Attribute(“x”))变成一个形如1+row.get(“x”)的scala表达式的AST。

Quasiquotes是编译时类型安全的，这确保了只有合适的AST或者常量才可以代替，使之比起字符串拼接更加便于使用，而且直接变为scala AST而不是在运行时运行scala转换器（parser）。此外，它们（Qua..）是高度组合的，因为每个节点代码生成的规则不需要知道已经构造好的它的孩子们是怎样被返回为树的。最后，以防万一有表达式级别的优化被Catalyst遗漏，scala编译器会进一步地优化，从而产生最终的代码。下面的图表展示了quasiquotes使我们生成了性能与手动调节的程序（hand-tuned programs）相当的代码。
